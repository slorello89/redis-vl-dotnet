## Codebase Patterns
- Antora docs live under `docs-site/`, use the local UI bundle in `docs-ui/`, and build the site from the repo root with `npm run docs:build`.
- GitHub Pages docs publishing uses `.github/workflows/docs-pages.yml`; keep the deploy job separate from the Antora build job and publish the `build/site` artifact only after a successful build.
- Keep docs and example quality gates aligned around repository-root commands: `npm run docs:validate` for Antora validation and `npm run examples:build` for the full example sweep, with CI calling the same entry points instead of duplicating shell one-liners.
- When Antora replaces legacy repository entry-point docs, move the substantive overview into the Antora section index pages and leave the old Markdown files as short redirect stubs so there is one canonical editing target.
- Keep connection and credential setup canonical in `docs-site/modules/ROOT/pages/getting-started/index.adoc`; examples can show story-specific overrides, but shared Redis topology and provider env var guidance should live in the getting-started page so links and onboarding stay consistent.
- For Antora pages under nested folders, use module-relative xref targets such as `xref:core-features/search-schema.adoc[]` instead of sibling-only filenames so local docs validation resolves links consistently.
- In Antora prose, avoid literal `{name}` placeholders unless they are real AsciiDoc attributes; use inline code like `prefix + id` or escape the braces so local docs validation does not emit missing-attribute warnings.
- For query and aggregation docs, pair `/examples/JsonStorageExample` and `/examples/VectorSearchExample` with `tests/RedisVlDotNet.Tests/Indexes/SearchQueryCommandBuilderTests.cs`, `tests/RedisVlDotNet.Tests/Indexes/SearchIndexAsyncTests.cs`, and `tests/RedisVlDotNet.Tests/Queries/SearchResultMappingTests.cs`; the examples show canonical flows while the tests capture normalization, paging, typed mapping, and runtime-vector edge cases.
- For higher-level cache and workflow docs, link runnable samples where they exist and pair them with unit/integration tests for the contract details; these APIs often hide important threshold, filter, TTL, and recency behavior in tests rather than in a single broad example.
- For provider extension docs, give each package its own Antora page under `docs-site/modules/ROOT/pages/extensions/`, document the required env var plus fail-fast missing-credential behavior, and point back to the matching smoke tests and runnable example README.
