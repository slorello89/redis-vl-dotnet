## Codebase Patterns

- Keep durable product and architecture contracts under `docs/`, and use the root `README.md` only as the short entry point into those deeper specs.
- Even for documentation-first iterations, keep a valid `.NET` solution and at least one buildable project in place so `dotnet build` remains a meaningful repository quality gate.
- Keep production code under `src/` and mirror public API coverage under `tests/`, with schema and other contract tests grouped by feature namespace so `dotnet test` validates repository progress as the surface area grows.
- Put schema guard clauses in the schema model constructors themselves so invalid configurations fail at object creation time and later Redis command builders can assume validated inputs.
- For file-based schema loading, deserialize into a small DTO layer that matches upstream YAML keys, then map into the validated public schema model so code-first and file-first flows share the same invariants.
- Run `dotnet test` and `dotnet build` sequentially in this repo; running them concurrently can contend on `tests/.../bin/Debug/net9.0/*.deps.json`.
