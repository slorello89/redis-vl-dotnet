= Core Features

This section is the canonical overview for the core Redis-backed library surface.

== Main workflow areas

The core package centers on these feature groups:

* xref:core-features/search-schema.adoc[`SearchSchema`] and xref:core-features/index-definition.adoc[`IndexDefinition`] for defining index names, prefixes, storage type, and fields
* xref:core-features/field-definitions.adoc[field definitions] for text, tag, numeric, geo, and vector indexing
* xref:core-features/index-lifecycle.adoc[`SearchIndex`] for index creation, inspection, listing, clearing, rediscovery, and dropping
* JSON and HASH document load, fetch, delete, and partial update flows
* structured query objects for filter, text, count, vector, hybrid, and aggregation workflows

== Primary example

The primary runnable example for this area is `/examples/JsonStorageExample`.
It demonstrates:

* loading a schema from YAML
* creating a JSON-backed index with overwrite semantics
* listing indexes and rediscovering one with `SearchIndex.FromExistingAsync`
* loading, fetching, partially updating, clearing, and dropping documents

Pair it with `/examples/VectorSearchExample` when you need the HASH-backed view of the same lifecycle and schema concepts.

== Storage behavior

The library supports both JSON and HASH storage models:

* `StorageType.Json` uses RedisJSON paths and requires RedisJSON plus RediSearch.
* `StorageType.Hash` uses Redis hashes and requires RediSearch only.

Choose the storage type in `IndexDefinition`, then use the matching `SearchIndex` APIs. JSON flows call methods such as `LoadJsonAsync` and `FetchJsonByIdAsync`; HASH flows call `LoadHashAsync` and `FetchHashByIdAsync`.

Query pagination is shared across the search APIs, and typed result mapping is designed to let callers work with strongly typed documents or projections instead of parsing raw Redis arrays directly.
