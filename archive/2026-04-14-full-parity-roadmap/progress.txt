## Codebase Patterns
- Keep parity-surface docs anchored by an examples feature map in `examples/README.md` and a command-level validation matrix in `docs/testing.md`, then validate doc/example updates by building every example project so discoverability and runnable coverage stay in sync.
- Keep the active roadmap contract in a dedicated doc and link it from `README.md` and `docs/getting-started.md` so implementation work and user-facing navigation stay aligned.
- When parity decisions depend on Java and Python comparisons, record intentional .NET-native differences explicitly so later feature work targets capabilities rather than cross-language API cloning.
- For index metadata extensions, preserve `IndexDefinition.Prefix` as the primary write-path namespace and add new FT.CREATE metadata via separate strongly typed properties so document loading APIs stay backward compatible.
- Model optional FT.CREATE metadata as nullable index properties where `null` means Redis defaults and an empty collection means an explicit `0`-count clause, so default and disabled behaviors remain distinguishable in both code-first and YAML schema paths.
- Emit RediSearch field modifiers in parser-safe order: `INDEXEMPTY`/`INDEXMISSING` before `SORTABLE`, `UNF` immediately after `SORTABLE`, and `NOINDEX` after sortable modifiers; vector `INDEXMISSING` belongs inside the vector attribute payload rather than after the field definition.
- Keep YAML schema loading strict: accept either `index.prefix` or `index.prefixes` for namespace configuration, map upstream-style underscored option names through DTOs, and fail fast on unsupported keys instead of silently ignoring them.
- Index maintenance helpers should operate on `Schema.Index.Prefixes` with batched Redis key scans and deletes so JSON and HASH storage share one implementation while the RediSearch index definition remains intact.
- Database-wide RediSearch discovery helpers should stay static on `SearchIndex` and map Redis array replies into small named DTOs, which keeps environment inspection APIs available without requiring a schema-bound instance.
- When rebuilding `SearchSchema` from `FT.INFO`, collapse JSON fields back to alias-less definitions only when the Redis `attribute` matches the default alias for the JSON path; otherwise preserve the raw identifier path plus explicit alias so recreated schemas round-trip cleanly.
- JSON partial update helpers should accept only absolute JSONPath expressions like `$.field` or `$[0]`, trim and deduplicate paths before issuing Redis commands, and treat missing root documents as a `false` return instead of implicitly creating new JSON keys.
- HASH partial update helpers should reuse a hash-exists probe plus a single `HSET` batch, accept only trimmed top-level field names, and reject `null` values because HASH writes preserve omitted fields rather than expressing field deletion.
- New FT.SEARCH query types should reuse the shared return-field normalization and the standard `RETURN`/`LIMIT`/`DIALECT 2` argument layout, which keeps query-specific additions isolated to the search string or extra clauses.
- FT.AGGREGATE query builders should normalize schema-backed property references to the runtime query field name while still allowing APPLY and REDUCE aliases to flow through as plain `@alias` references in later pipeline stages.
- Redis-backed query integration tests should seed tiny HASH datasets with intentionally different match strength, wait on `num_docs` before asserting, and verify projected-field omission explicitly when a query narrows `RETURN` fields.
- Typed aggregation projections should use reducer aliases that match the serializer naming policy used by `SearchResultMapper` (for example `movieCount` instead of `movie_count`) unless the target properties add explicit `JsonPropertyName` attributes.
- Aggregate hybrid support should reuse the plain aggregation pipeline model and treat the KNN score alias like any other aggregation property reference, so reducers and sort clauses can operate on `vector_distance` without a separate result-mapping path.
- Multi-vector search on the current Redis 7.4 test target should compose existing per-field `FT.SEARCH` calls, restrict inputs to cosine vector fields, and merge only intersection candidates using weighted distance ordering plus stable per-clause and document-id tie-breaks.
- Runtime vector tuning should stay query-local and flow through the shared vector command builder so composed queries reuse one `PARAMS` serializer while schema-based algorithm checks reject HNSW-only options on `FLAT` fields before Redis execution.
- Shared query paging should flow through `QueryPagination`; vector-style queries keep `topK` as the retrieval window and must reject `offset + limit` values that would page past that window.
- Batch query iterators should reuse the existing query execution paths by cloning the original query with a new `QueryPagination` each round; vector-style batches must clamp the final page to the remaining `topK` window instead of widening it.
- Recency-based workflow storage should use Redis-side monotonic sequence counters per logical scope (for example, per session) and sort on that sequence, which avoids nondeterministic ordering when caller-supplied timestamps collide.
- Semantic workflow retrieval should keep session scoping in the RediSearch filter and express similarity as a `VectorRangeQuery` distance threshold, which lets role filters compose cleanly while preserving deterministic recency metadata on the stored HASH documents.
- Semantic cache filter variants should declare explicit HASH-backed TAG/TEXT/NUMERIC fields in the cache options, validate supplied filter values against that schema before writes, and fold the normalized filter payload into the cache key so prompt variants can coexist without collisions.
- Provider-backed embedding integrations should live in a separate abstractions package plus optional provider packages; keep the core library consuming the shared vectorizer interfaces and retain any legacy core-owned embedding interfaces only as migration shims.
- Provider extension packages should accept the vendor SDK client as an injectable dependency for unit tests, keep any live provider coverage behind env-gated smoke tests, and avoid exposing SDK members marked as evaluation-only on the public wrapper surface.
- HTTP-based provider vectorizers should model the provider's published JSON payload directly, omit unset optional fields from serialized requests, and allow an endpoint override so shared tests can assert request shape without depending on one hosted route.
- HTTP-based reranker integrations should send only provider-supported document text in the payload, then remap provider result indexes back onto the original `RerankDocument` instances locally so caller metadata survives reranking without custom server-side schemas.
- AI extension seams stay easier to evolve when the abstractions assembly owns immutable request/result DTOs and convenience extensions, while future provider packages enter the solution first as marker scaffolds that reference only the shared abstractions package.
- Redis topology helpers should stay outside `SearchIndex` and return plain `ConfigurationOptions` or `IConnectionMultiplexer`, so cluster and Sentinel support can evolve independently while the rest of the library continues to consume `IDatabase`.
- Sentinel helpers should build one reusable `ConfigurationOptions` object with `CommandMap.Sentinel`, a required `ServiceName`, default host-only node port `26379`, and a separate primary-connection helper that reuses that same option shape for failover-aware discovery.
- CLI workflows stay easiest to test when argument parsing produces immutable request records and one Redis-backed service layer handles the actual `SearchIndex` calls, so command help/output can be unit-tested in-process while Redis integration coverage reuses the existing env-gated harness.
- For YAML-backed CLI flows, validate schema files while constructing the immutable request and pass the realized `SearchSchema` through the service layer, so `schema` commands and `index create --schema` stay on one parsing path.
