## Codebase Patterns

- Keep durable product and architecture contracts under `docs/`, and use the root `README.md` only as the short entry point into those deeper specs.
- Even for documentation-first iterations, keep a valid `.NET` solution and at least one buildable project in place so `dotnet build` remains a meaningful repository quality gate.
- Keep production code under `src/` and mirror public API coverage under `tests/`, with schema and other contract tests grouped by feature namespace so `dotnet test` validates repository progress as the surface area grows.
- Put schema guard clauses in the schema model constructors themselves so invalid configurations fail at object creation time and later Redis command builders can assume validated inputs.
- For file-based schema loading, deserialize into a small DTO layer that matches upstream YAML keys, then map into the validated public schema model so code-first and file-first flows share the same invariants.
- Run `dotnet test` and `dotnet build` sequentially in this repo; running them concurrently can contend on `tests/.../bin/Debug/net9.0/*.deps.json`.
- Keep Redis-backed integration tests opt-in via `REDIS_VL_REDIS_URL` and a custom xUnit `Fact` attribute so the default test run stays green on machines without Redis Stack.
- For `ON JSON` index creation, translate schema field names into JSONPath identifiers and default the RediSearch alias to the logical field name so the public schema model does not force callers to write raw JSONPath everywhere.
- For JSON document lifecycle APIs, resolve Redis keys from an explicit key first, then an explicit or selected id, and finally a conventional public `Id` property so single-document and batch loads stay on one predictable key contract.
- For `ON HASH` document lifecycle APIs, serialize POCOs with the shared web-default JSON naming policy so Redis hash fields stay camel-cased like schema field names, then rehydrate typed documents from `HashEntry[]` using the same property-name contract.
- Keep RediSearch query DSL features as immutable expression objects that render query fragments directly, with `And`/`Or`/`Not` helpers and operator overloads as thin composition sugar rather than embedding string concatenation in later query APIs.
- Keep RediSearch command shaping and raw response parsing in dedicated internal query builders and parsers so public query objects stay schema-focused and unit tests can validate command contracts without a live Redis connection.
- For non-vector retrieval and count flows, keep using `FT.SEARCH` on the shared parser path; use `NOCONTENT LIMIT 0 0` for counts so filter, vector, and count abstractions all share one result-shaping pipeline.
- Keep vector-derived query types responsible for normalizing their distance alias into `RETURN` fields, and enforce hybrid-query lexical intent by requiring at least one text predicate while still composing extra metadata filters through the shared filter DSL.
- Layer typed search materialization on top of raw `SearchResults` with the shared web-default JSON serializer, and treat the Redis document id as a synthetic `id` field so projections can map into DTOs without a second query pipeline.
- For async Redis-backed APIs, throw on cancellation before issuing direct `IDatabase` calls or doing expensive serialization/mapping work, and cover that behavior with unit tests that use a `DispatchProxy` `IDatabase` recorder instead of a live server.
- For exact-match cache primitives, use a dedicated Redis keyspace with per-entry TTL, hash user input into bounded keys, and store the original input alongside the payload so collision checks preserve exact-match semantics.
- For semantic cache primitives, back each cache with a dedicated RediSearch index over HASH documents, store vector payloads as raw bytes in hash fields, and hash prompts into stable Redis keys so semantic lookup and per-entry TTL can coexist cleanly.
- For higher-level semantic workflows, layer on `SearchIndex` with dedicated HASH-backed workflow documents and write vector bytes directly into hash fields so provider-agnostic routing or cache APIs can reuse the query stack without depending on the generic POCO hash mapper.
- For Redis-backed integration coverage, keep deterministic seed datasets in shared test fixtures and wait on index readiness with bounded polling helpers instead of fixed `Task.Delay(...)` sleeps.
- Keep end-to-end onboarding docs in `docs/` with copy-pasteable commands and API-accurate snippets, and use the root `README.md` as a short navigator into those longer guides.
- Add directory-level `README.md` files when a repo area grows multiple runnable entry points, and cross-link those indexes from the root `README.md`, nearby guides, and leaf example docs so navigation mirrors the repository structure.
- Keep runnable samples as checked-in console apps under `examples/` with direct project references to `src/RedisVlDotNet`, and include them in the solution so `dotnet build` validates example drift alongside the library.
- For HASH-backed vector fixtures and examples, seed vector fields with raw float32 bytes via direct `HashEntry` writes and wait for index readiness before issuing nearest-neighbor queries.
