## Codebase Patterns
- `EmbeddingsCache` compatibility depends on leaving the legacy input-only key path unchanged; add parity features through new overloads or optional identity components rather than altering `CreateKey(input)` behavior.
- Hash-backed cache metadata in this repo is stored as a JSON string serialized with `JsonSerializerDefaults.Web`; richer read models should preserve that payload verbatim instead of trying to deserialize it eagerly.
- For `EmbeddingsCache` parity work, keep `Store`/`Lookup` as compatibility aliases and layer newer Python-style `Set`/`Get` entry-centric APIs on top so callers can adopt richer results without breaking existing code.
- `EmbeddingsCache` TTL precedence should be `per-call override ?? cache default`; unit tests can verify this by recording `KeyExpireAsync` arguments in the existing `DispatchProxy` database fake.
- When adding more `EmbeddingsCache` read variants, route them through a shared hash-to-entry parser so semantic lookups can keep validating input/model identity while direct-key APIs reuse the same decode path.
- Single-entry `EmbeddingsCache` lifecycle methods should keep returning `bool` hit/miss semantics from Redis single-key commands (`KeyExistsAsync`, `KeyDeleteAsync`) so future batch APIs can reserve count-based returns for multi-key operations.
- Batch `EmbeddingsCache` APIs should preserve caller order and represent misses in-place (`null` for reads, `false` for existence) so mixed hit/miss workflows do not need an additional join step.
- `EmbeddingsCache` docs in this repo should keep the Antora page, example `Program.cs`, and example `README.md` in sync so API parity changes are reflected in both narrative docs and runnable usage.
